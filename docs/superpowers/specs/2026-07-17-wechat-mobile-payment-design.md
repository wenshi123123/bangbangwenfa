# 微信移动端支付设计

## 目标

在同一网站支付入口中，根据用户访问环境完成正确的微信支付：微信内使用 JSAPI，手机外部浏览器使用 H5 支付，电脑浏览器使用 Native 扫码支付。

## 前提

- 微信支付商户号已认证，且已绑定已认证服务号 AppID。
- 公众号网页授权域名、JSAPI 支付授权目录和 H5 支付域名均已审核并指向正式 HTTPS 域名。
- 微信支付 API v3 证书、商户私钥、API v3 密钥、公众号 AppID/Secret 均只保存在服务端环境变量中。

## 支付分流

| 访问环境 | 下单接口 | 前端动作 |
| --- | --- | --- |
| 微信客户端内 | JSAPI | 使用 `WeixinJSBridge.invoke('getBrandWCPayRequest')` 拉起支付 |
| 手机外部浏览器 | H5 | 跳转微信返回的 `h5_url`，附带 `redirect_url` |
| 电脑浏览器 | Native | 展示 `code_url` 二维码 |

微信内绝不创建 H5 订单，也不降级为 Native 二维码；若未获得可用的微信用户标识，先进行公众号 OAuth，再继续创建 JSAPI 订单。

## OAuth 与支付会话

1. 支付页识别到微信客户端且没有支付会话时，跳转至 `/api/wechat/oauth/authorize`。
2. OAuth 使用 `snsapi_base`，回调由服务端以授权 code 换取 `openid`。
3. 回调不再把 `openid` 放入 URL 或 localStorage；它写入 HTTP-only、Secure、SameSite=Lax 的短时支付会话 Cookie，再回到原支付页。
4. 创建支付订单接口从该会话读取 `openid`，并仅在微信内创建 JSAPI 预支付订单。
5. 回调前保留原始目标路径和允许的查询参数；只接受站内相对路径，避免开放重定向。

## 后端边界

- 抽出请求环境识别：`wechat`、`mobile`、`desktop`。
- 咨询订单和律师入驻订单共用该识别规则与 OAuth 会话读取方式。
- 服务端生成 JSAPI 签名参数；私钥、API v3 密钥和公众号 Secret 永不返回客户端。
- 订单归属校验保持有效：已登录用户按用户 ID 校验；微信 OAuth 访问仅接受与订单绑定的 `openid`。
- 订单“已支付”只能由微信支付异步通知验签、解密、金额核验和幂等更新后写入。浏览器回调仅用于展示结果。

## 前端行为

- 咨询支付页与律师入驻支付页均在微信内先确保 OAuth 会话存在，再请求创建订单。
- JSAPI 成功、取消、失败分别展示明确状态；成功后查询订单状态，确认服务端回调已落库后跳转成功页。
- H5 支付返回后也查询订单状态；如果回调尚未到达，显示处理中并短暂重试。
- 预支付创建和 JSAPI 拉起避免重复触发，防止重复订单。

## 验收与测试

- 在 iOS、Android 微信客户端真机内，可完成 OAuth、拉起 JSAPI、收到回调并进入成功页。
- 在 iOS Safari、Android Chrome 等外部浏览器中，可跳转 H5 支付并回站。
- 在桌面浏览器中可显示 Native 二维码。
- 验证取消支付、支付失败、网络中断、重复点击、微信支付回调重试、重复回调和金额不匹配均不会误将订单设为已支付。

## 非目标

- 本次不接入小程序支付、APP 支付、退款或分账。
- 本次不改变既有订单价格、支付回调业务规则或商户配置。
