# Harness Engineering 说明

这份文档是本仓库中用于 AI 自主开发任务的统一入口。目标不是增加流程负担，而是把“地图、目标、验收标准”提前写清楚，减少反复沟通和跑偏。

## 适用范围

- 新功能开发
- 缺陷修复
- 发布前验证
- 需要交给 AI worker 或子代理拆分执行的任务

## 项目地图

| 模块 | 位置 | 说明 |
| --- | --- | --- |
| Web 应用 | `src/app` | Next.js App Router 页面、路由和 API。 |
| 组件层 | `src/components` | 公共 UI、业务组件和流程组件。 |
| 认证 | `src/lib/auth`、`src/hooks/use-auth.tsx` | 登录态、JWT、守卫和管理员认证。 |
| 支付 | `src/lib/payment`、`src/app/api/pay` | 微信支付、回调和相关页面。 |
| 短信 | `src/lib/sms` | 验证码、通知和发送适配。 |
| 数据库 | `src/storage/database`、`scripts/*.sql` | Supabase / PostgreSQL 相关 schema 和迁移脚本。 |
| 运维与验证 | `scripts`、`docs`、`e2e` | 构建、部署、验收、生产只读 smoke 和回归测试。 |

## 任务目标格式

每个任务都应该先写成下面这种结构：

```md
## 任务目标
- 要做什么：
- 为什么要做：
- 不做什么：
- 影响范围：
```

要求：

- 目标必须具体，避免“优化一下”“顺手修掉”这种模糊描述。
- 必须明确不在范围内的内容，防止任务膨胀。
- 如果任务会改动多个模块，先写出主路径和副路径。

## 验收标准格式

每个任务必须带可验证的完成标准：

```md
## 验收标准
- [ ] `pnpm ts-check` 通过
- [ ] `pnpm build` 通过
- [ ] `pnpm test:prod:smoke` 通过，或明确说明为什么不需要
- [ ] 涉及登录 / 支付 / 后台 / 律师入口时，补充真实链路验证
- [ ] 关键页面、接口或脚本行为与预期一致
```

如果任务不需要全部命令，也要说明原因，例如：

- 只改文档时，`pnpm build` 可能不是主验证项，但仍要确认不破坏相关引用。
- 只改脚本时，至少要运行对应脚本和受影响的检查命令。

## 风险与约束清单

每次开始前都要检查这些内容：

- 是否会改动生产路径或线上默认行为
- 是否涉及支付、登录、权限、数据写入
- 是否会影响数据库 schema、迁移或环境变量
- 是否会暴露密钥、token、诊断接口或调试接口
- 是否需要兼容现有部署方式和云托管启动方式
- 是否存在只读验证不足以覆盖的真实业务风险

## 什么时候拆成多个代理

满足下面任一条件时，优先拆分：

- 涉及两个以上彼此独立的代码区域
- 既要改实现，又要补验证
- 任务同时包含调研、实现、回归修复
- 需要并行处理前端、后端、脚本或测试

推荐分工方式：

- 1 个代理负责调研和约束梳理
- 1 个代理负责主要实现
- 1 个代理负责验证和回归检查

拆分时要保证：

- 每个代理有明确 ownership
- 每个代理的写入范围尽量不重叠
- 主代理负责整合最终结果，不重复造轮子

## 推荐工作流

1. 先写任务卡，补齐地图、目标、验收标准。
2. 再决定是否需要拆分代理。
3. 完成实现后，按验收标准逐项验证。
4. 只有在证据齐备后，才宣布完成。
